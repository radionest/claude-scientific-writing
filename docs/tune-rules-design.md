# Design: `tune-rules` — калибровка правил плагина по правкам пользователя

**Date:** 2026-06-24
**Status:** approved design (brainstorm), pre-plan
**Companion:** `docs/design.md` (мастер-дизайн плагина; severity-политика — там же, § «Severity policy»)

## Goal

Замкнуть петлю обратной связи плагина. Сейчас направление одно: правила → применяются к тексту
(`scientific-writer`, линтер, прозо-скиллы). Эта возможность строит обратное: **правка текста
пользователем → изменения правил**. Заменяет ручной сбор калек в словарь (прежние `feedback_*`-мемори) на
воспроизводимую команду, которая ещё и доказывает предложенное правило на тест-оракуле.

**Принцип сигнала (плагин проверяется на собственном выходе).** «До» — гарантированно чистый по линтеру
выход `scientific-writer`. Поэтому любая правка пользователя — это то, чего правила **не ловят** (false
negative) либо что они **переусердствовали** ловить (false positive). Шум наивного LLM исключён заведомо.

## Решения (из обсуждения)

| Вопрос | Решение |
|---|---|
| Источник «до» | генерация из источника с применением канона (writer применяет канон + само-линт) |
| Глубина выхода | правки + самопроверка словаря (fixture + pytest до применения); скилл/канон — правка на подтверждение |
| Масштаб | каждый запуск независим; **без** корпуса/состояния между запусками |
| Движок вывода | главный агент под скиллом `tuning-rules` + тонкий детерминированный вспомогательный модуль (`tune_diff.py`) |
| Поверхность | **две команды**: `tune-draft` (генерация) + `tune-learn` (вывод правил) |

## Поверхность команд

Две фазы с паузой на ручную правку.

### `commands/tune-draft.md` — генерация
`/scientific-writing:tune-draft <source> [genre]`
- `<source>` = URL | тема-строка | путь к тексту/.qmd. `genre` ∈ {report, article, educational};
  не задан → спросить (default: article).
- Шаги:
  1. Определить источник: URL → загрузить; файл → прочитать; тема → задание. Необязательная подсказка длины (напр. «~1000 слов»).
  2. Вызвать агента `scientific-writer` в режиме «tuning-fixture» (**без** `<doc>.spec.md`): черновик,
     соответствующий канону, из источника по профилю жанра + канону + словарю. Спецификация намеренно
     пропускается — это калибровочная фикстура, а не документ проекта, поэтому правило «нет документа без
     спецификации» (из `writing-scientific-documents`) здесь не действует.
  3. **Само-линт-цикл** (≤3 итерации): `PYTHONPATH=${CLAUDE_PLUGIN_ROOT} python3 -m lib.lint_prose <draft> --json`
     → писатель исправляет error/warn-находки → повтор, пока чисто (или только `lint-ok`-подавлено).
     Цель: «до» гарантированно чистый по линтеру. Не свелось за 3 → отдать с пометкой (остаточные находки — тоже сигнал).
  4. Записать артефакты в стабильный tmp-каталог (default `/tmp/sci-tune/`, имена соседних файлов выводятся из основного):
     - `<slug>.qmd` — черновик (его правит пользователь на месте);
     - `<slug>.pristine.qmd` — нетронутая копия (сохранённая копия);
     - `<slug>.tune.json` — файл-спутник (см. схему ниже).
  5. Печать: путь к `<slug>.qmd` + «правь вручную, потом `/scientific-writing:tune-learn <slug>.qmd`».

Файл-спутник `<slug>.tune.json`:
```json
{"source": "<url|topic|path>", "genre": "article",
 "draft": "<slug>.qmd", "pristine": "<slug>.pristine.qmd", "lint_clean": true}
```

### `commands/tune-learn.md` — вывод правил
`/scientific-writing:tune-learn <edited.qmd> [--pristine <path>]`
- Определение «до»: pristine + файл-спутник как соседние файлы от `<edited.qmd>` (имена выводятся из основного).
  Параметр `--pristine` → заодно покрывает случай «приношу готовую пару» (генерация не обязательна).
- Шаги:
  1. Вспомогательный модуль `tune_diff.py` → прозо-фрагменты pristine→edited + FN/FP-разметка (контракт ниже).
     Прозы-правок нет → «нечего выводить», остановиться.
  2. Агент под скиллом `scientific-writing:tuning-rules` читает размеченные фрагменты + `canon/dictionary.json` +
     оба прозо-скилла + профиль жанра + (если есть) `<doc>.spec.md`. Классифицирует каждый фрагмент по таксономии,
     маршрутизирует, пишет правку.
  3. Самопроверка словаря (для каждой записи словаря — процедура ниже). Регресс/FP → правку **не применять**.
  4. Отчёт + правки на подтверждение: сгруппировано по слою, каждая с (a) фрагментом-обоснованием, (b) классом,
     (c) правкой, (d) для словаря — зелёным тестом.
  5. Подтверждение → применить в рабочее дерево (Edit/Write), фикстуры оставить (они — доказательство правила),
     прогнать весь `pytest`. **Не делать коммит** — пользователь проверяет и делает коммит сам.

`plugin.json` не трогаем (команды/скиллы обнаруживаются автоматически). Хук `text-lint.sh` уже пропускает
`tests/fixtures/*.qmd` (коммит 244a904) → добавление фикстур-строк не самоблокируется при коммите.

## `lib/tune_diff.py` — контракт

- CLI: `PYTHONPATH=${CLAUDE_PLUGIN_ROOT} python3 -m lib.tune_diff <pristine.qmd> <edited.qmd> [--dictionary PATH] --json`
- Логика:
  1. `extract_prose` на обоих файлах (пропуск кода/yaml/таблиц/цитат — как в линтере).
  2. Выровнять прозо-блоки (`difflib.SequenceMatcher` по блокам), взять `replace/insert/delete`-опкоды.
  3. Для каждого изменённого блока прогнать regex словаря (`load_entries`):
     - `after` ловит правило → **FP-кандидат** (пользователь ушёл к тому, что правило запрещает), записать `rule_hit`;
     - `after` чист, текст изменился → **FN-кандидат** (агент решит: калька / суждение / шум);
     - `before` ловит правило (контроль — pristine должен быть чист) → пометить `before_dirty` (промах само-линта).
- Выход JSON: `[{"line": int, "before": str, "after": str, "triage": "FN|FP|neutral", "rule_hit": "id|null", "before_dirty": bool}]`
- Stdlib + переиспользование `lib/qmd_prose.py`, `lib/lint_prose.py`. Без новых зависимостей.

## Таксономия и маршрутизация (ядро `tuning-rules`)

| Класс правки | Маршрут | Самопроверка |
|---|---|---|
| Пропущенная калька (FN), regex-выразима | новая/расширенная запись `canon/dictionary.json` | fixture + pytest |
| Перекомпенсация (FP) — правка трогает правило | `except` / сузить `pattern` / `severity`↓ / уточнение в скилле | pytest без регресса |
| Суждение (фраза, regex не выражает) | строка в таблице `skills/writing-russian-academic-prose` | — (на подтверждение) |
| Документный (повтор/термин-дрейф) | `skills/say-it-once-academic-prose` | — |
| Структура/жанр | `canon/profiles/<жанр>.md` | — |
| Терм конкретного документа | `<doc>.spec.md` — **не** глобальное правило | — |
| Шум / разовый вкус | ничего (показать как «не обобщаю») | — |
| Нужен код | regex не выражает → правка логики `lib/lint_prose.py` | существующие тесты |

**Фильтр шума** (в скилле): одиночная стилевая правка без обобщаемого паттерна и терм, относящийся к
конкретному документу, — не глобальное правило. Финальный фильтр генеральности — подтверждение/отклонение пользователя.

**Порядок ослабления при FP** (в скилле): сначала `except` (узкий допустимый токен) → затем сузить `pattern`
(граница `\b`, уточнить контекст) → затем `severity`↓ (error→warn) → в крайнем случае уточнение в прозо-скилле.

## Дисциплина словаря (самопроверка перед применением)

Процедура зависит от класса правки (детали — в скилле `tuning-rules`):
- **Новое правило (FN):** добавить убранную фразу в `tests/fixtures/calques.qmd` (оракул) → `pytest`
  (`test_dictionary` + `test_lint_prose`) доказывает: правило ловит фразу, нет регресса, `id` уникальны, схема
  валидна; затем `lint_prose <edited.qmd>` — новое правило не даёт находку на **оставленном хорошем тексте**.
- **Ослабление правила (FP):** оракул **не** трогать (оставленный текст — не калька, иначе сломается
  `test_every_fixture_phrase_is_flagged`). `lint_prose <edited.qmd>` → у правила больше нет находки с
  `severity: error` (`except`/сужение убирают её; `severity↓` оставляет `warn`); `pytest` зелёный — ослабление
  не сломало выявление истинных калек.
Любой шаг провалился → правку **не применять**.

Severity-политика: `docs/design.md` § «Severity policy» (error — однозначные лексические кальки, низкий FP-риск;
warn — эвристики, FP-prone).

## Поток данных

```
source (URL/тема/файл)
   │  /tune-draft
   ▼
scientific-writer → draft.qmd → lint_prose (само-исправление ≤3) → чистый по линтеру
   ├─ pristine.qmd  (сохранённая копия)
   └─ draft.qmd     (правишь вручную)  ◄──── [ОФЛАЙН: ручная правка]
                                                │  /tune-learn draft.qmd
                                                ▼
              tune_diff.py: прозо-фрагменты + FN/FP-разметка (regex словаря vs edited)
                                                ▼
              tuning-rules агент: классифицирует → маршрутизирует → пишет правки
              ┌─────────────────────────────────┼─────────────────────────────┐
              ▼                                  ▼                             ▼
      dictionary.json запись              скилл/канон/профиль            lib код-правка
       + строка-фикстура                        │                             │
       + pytest (доказать + без регресса)       │                             │
              └──────────────► ОТЧЁТ + ПРАВКИ (подтверждение) ◄───────────────┘
                                                │ подтверждение
                                                ▼
                            применить в рабочее дерево + полный pytest зелёный
                                       (пользователь проверяет + делает коммит)
```

## Обработка ошибок

- `tune-draft`: источник недоступен → сообщить, спросить альтернативу. Само-линт не свёлся за 3 итерации →
  отдать с пометкой. Утверждение нельзя привести в соответствие с каноном → `<!-- TODO ред: … -->` (как сейчас в writer).
- `tune-learn`: нет файла-спутника/pristine → ошибка + подсказка `--pristine`. Прозы-правок нет → «нечего выводить».
  Regex ломает тест либо даёт FP на хорошем тексте → эту правку **не применять**, показать провал; агент
  сужает паттерн или переводит правку в предложение для скилла.

## Тестирование

- `tests/test_tune_diff.py` — фикстура-пара before/after → ожидаемая разметка (FN/FP/neutral + `rule_hit`).
- Самопроверка словаря — рантайм-доказательство предлагаемых правил (не коммит-тест).
- Шаг применения `tune-learn` прогоняет весь `pytest` (проверка на регресс).

## Вне области (YAGNI)

Корпус/состояние между запусками; ансамбль субагентов; авто-коммит/рендеринг; пороги авто-продвижения по
повторяемости; итеративное совместное письмо (генерация — один проход + само-линт). Не-русская проза. Изменение
механики docx-импорта.

## Открытые вопросы

- Выравнивание прозо-блоков в `tune_diff.py` при крупных перестановках абзацев — `difflib` по блокам может
  дать зашумлённые `replace`; если на реальных правках зашумлено, перейти на якоря по неизменным предложениям.
  Решать по первым запускам, не усложнять заранее.
- Стабильность tmp-пути между сессиями (`tune-draft` и `tune-learn` могут идти в разных сессиях) — решено
  именами соседних файлов, выводимыми из основного, + параметром `--pristine`.
- `triage="neutral"` объединяет чистую вставку (`before` пуст) и чистое удаление (`after` пуст) — обе слабый
  сигнал, агент судит сам. Для v1 достаточно; при необходимости развести по пустой стороне (`insert`/`delete`).
