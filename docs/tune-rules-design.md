# Design: `tune-rules` — калибровка правил плагина по правкам пользователя

**Date:** 2026-06-24
**Status:** approved design (brainstorm), pre-plan
**Companion:** `docs/design.md` (мастер-дизайн плагина; severity-политика — там же, строки 98–110)

## Goal

Замкнуть петлю обратной связи плагина. Сейчас направление одно: правила → применяются к тексту
(`scientific-writer`, линтер, прозо-скиллы). Эта фича строит обратное: **правка текста пользователем →
дельты правил**. Заменяет ручной майнинг калек в словарь (прежние `feedback_*`-мемори) на
воспроизводимую команду, которая ещё и доказывает предложенное правило на тест-оракуле.

**Принцип сигнала (канон-чистый дог-фудинг).** «До» — гарантированно linter-clean выход
`scientific-writer`. Поэтому любая правка пользователя — это то, чего правила **не ловят** (false
negative) либо что они **переусердствовали** ловить (false positive). Шум наивного LLM исключён по
построению.

## Решения (из брейншторма)

| Вопрос | Решение |
|---|---|
| Источник «до» | канон-чистая генерация из источника (writer применяет канон + само-линт) |
| Глубина выхода | патчи + самопроверка словаря (fixture + pytest до применения); скилл/канон — патч на approve |
| Масштаб | каждый прогон независим; **без** корпуса/состояния между прогонами |
| Движок вывода | главный агент под скиллом `tuning-rules` + тонкий детерминированный хелпер (`tune_diff.py`) |
| Поверхность | **две команды**: `tune-draft` (генерация) + `tune-learn` (вывод правил) |

## Поверхность команд

Две фазы с офлайн-паузой на ручную правку.

### `commands/tune-draft.md` — генерация
`/scientific-writing:tune-draft <source> [genre]`
- `<source>` = URL | тема-строка | путь к тексту/.qmd. `genre` ∈ {report, article, educational};
  не задан → спросить (default: article).
- Шаги:
  1. Резолвить источник: URL → fetch; файл → read; тема → бриф. Опц. подсказка длины (напр. «~1000 слов»).
  2. Дёрнуть агента `scientific-writer`: канон-чистый черновик из источника по профилю жанра + канону + словарю.
  3. **Само-линт-цикл** (≤3 итерации): `PYTHONPATH=${CLAUDE_PLUGIN_ROOT} python3 -m lib.lint_prose <draft> --json`
     → писатель фиксит error/warn-находки → повтор, пока чисто (или только `lint-ok`-подавлено).
     Цель: «до» гарантированно linter-clean. Не сошлось за 3 → отдать с пометкой (остаточные находки — тоже сигнал).
  4. Записать артефакты в стабильный tmp-каталог (default `/tmp/sci-tune/`, derived sibling-имена):
     - `<slug>.qmd` — черновик (его правит пользователь на месте);
     - `<slug>.pristine.qmd` — нетронутая копия (стэш);
     - `<slug>.tune.json` — сайдкар (см. схему ниже).
  5. Печать: путь к `<slug>.qmd` + «правь руками, потом `/scientific-writing:tune-learn <slug>.qmd`».

Сайдкар `<slug>.tune.json`:
```json
{"source": "<url|topic|path>", "genre": "article",
 "draft": "<slug>.qmd", "pristine": "<slug>.pristine.qmd", "lint_clean": true}
```

### `commands/tune-learn.md` — вывод правил
`/scientific-writing:tune-learn <edited.qmd> [--pristine <path>]`
- Резолв pristine+сайдкара как siblings от `<edited.qmd>` (derived-имена). `--pristine` override →
  бесплатно покрывает кейс «приношу готовую пару» (генерация не обязательна).
- Шаги:
  1. Хелпер `tune_diff.py` → прозо-хунки pristine→edited + FN/FP-триаж (контракт ниже). Прозы-правок нет → «нечего выводить», стоп.
  2. Агент под скиллом `scientific-writing:tuning-rules` читает триаж-хунки + `canon/dictionary.json` +
     оба прозо-скилла + профиль жанра + (если есть) `<doc>.spec.md`. Классифицирует каждый хунк по таксономии,
     маршрутизирует, пишет патч.
  3. Самопроверка словаря (для каждой dictionary-записи — процедура ниже). Регресс/FP → патч **не применять**.
  4. Отчёт + патчи на approve: сгруппировано по слою, каждый с (a) хунк-евиденс, (b) класс, (c) патч,
     (d) для словаря — зелёный тест.
  5. Approve → применить в рабочее дерево (Edit/Write), фикстуры оставить (они — доказательство правила),
     переганять весь `pytest`. **Не коммитить** — пользователь ревьюит и коммитит сам.

## `lib/tune_diff.py` — контракт

- CLI: `PYTHONPATH=${CLAUDE_PLUGIN_ROOT} python3 -m lib.tune_diff <pristine.qmd> <edited.qmd> [--dictionary PATH] --json`
- Логика:
  1. `extract_prose` на обоих файлах (игнор кода/yaml/таблиц/цитат — как в линтере).
  2. Выровнять прозо-блоки (`difflib.SequenceMatcher` по блокам), взять `replace/insert/delete`-опкоды.
  3. Для каждого изменённого блока прогнать regex словаря (`load_entries`):
     - `after` ловит правило → **FP-кандидат** (пользователь ушёл к тому, что правило запрещает), записать `rule_hit`;
     - `after` чист, текст изменился → **FN-кандидат** (агент решит: калька / суждение / шум);
     - `before` ловит правило (sanity — pristine должен быть чист) → пометить `before_dirty` (промах само-линта).
- Выход JSON: `[{"para": int, "before": str, "after": str, "triage": "FN|FP|neutral", "rule_hit": "id|null", "before_dirty": bool}]`
- Stdlib + переиспользование `lib/qmd_prose.py`, `lib/lint_prose.py`. Без новых зависимостей.

## Таксономия и маршрутизация (ядро `tuning-rules`)

| Класс правки | Маршрут | Самопроверка |
|---|---|---|
| Пропущенная калька (FN), regex-выразима | новая/расширенная запись `canon/dictionary.json` | fixture + pytest |
| Перекомпенсация (FP) — правка трогает правило | `except` / сузить `pattern` / `severity`↓ / оговорка в скилл | pytest без регресса |
| Суждение (фраза, regex не выражает) | строка в таблице `skills/writing-russian-academic-prose` | — (на approve) |
| Документный (повтор/термин-дрейф) | `skills/say-it-once-academic-prose` | — |
| Структура/жанр | `canon/profiles/<жанр>.md` | — |
| Терм конкретного документа | `<doc>.spec.md` — **не** глобальное правило | — |
| Шум / разовый вкус | ничего (показать как «не обобщаю») | — |
| Нужен код | regex не выражает → правка логики `lib/lint_prose.py` | существующие тесты |

**Шумофильтр** (в скилле): одиночная стилевая правка без обобщаемого паттерна и доку-специфичный терм —
не глобальное правило. Финальный фильтр генеральности — approve/reject пользователя.

**FP-плейбук** (в скилле): сначала `except` (узкий легитимный токен) → затем сузить `pattern` (граница
`\b`, уточнить контекст) → затем `severity`↓ (error→warn) → в крайнем случае оговорка-нюанс в прозо-скилл.

## Дисциплина словаря (самопроверка перед применением)

Для каждой новой/изменённой записи `canon/dictionary.json`:
1. добавить убранную пользователем фразу строкой в `tests/fixtures/calques.qmd` (оракул);
2. `PYTHONPATH=. python3 -m pytest tests/test_dictionary.py tests/test_lint_prose.py` → доказать:
   (i) новое правило ловит фразу, (ii) нет регресса (все прежние фикстуры ловятся, `id` уникальны, схема валидна);
3. прогнать новый regex против **оставленного хорошего текста** правки (поле `after`) — ловит → ложное
   срабатывание → сузить/в скилл, **не применять**;
4. показать зелёный тест в отчёте.

Severity-политика: `docs/design.md` строки 98–110 (error — однозначные лексические кальки, низкий FP-риск;
warn — эвристики, FP-prone).

## Поток данных

```
source (URL/тема/файл)
   │  /tune-draft
   ▼
scientific-writer → draft.qmd → lint_prose (само-фикс ≤3) → linter-clean
   ├─ pristine.qmd  (стэш)
   └─ draft.qmd     (правишь на месте)  ◄──── [ОФЛАЙН: ручная правка]
                                                │  /tune-learn draft.qmd
                                                ▼
              tune_diff.py: прозо-хунки + FN/FP-триаж (regex словаря vs edited)
                                                ▼
              tuning-rules агент: классифицирует → маршрутизирует → пишет патчи
              ┌─────────────────────────────────┼─────────────────────────────┐
              ▼                                  ▼                             ▼
      dictionary.json запись              скилл/канон/профиль            lib код-правка
       + строка-фикстура                        │                             │
       + pytest (доказать + без регресса)       │                             │
              └──────────────► ОТЧЁТ + ПАТЧИ (approve) ◄──────────────────────┘
                                                │ approve
                                                ▼
                            применить в рабочее дерево + полный pytest зелёный
                                       (пользователь ревьюит + коммитит)
```

## Обработка ошибок

- `tune-draft`: источник недоступен → сообщить, спросить альтернативу. Само-линт не сошёлся за 3 итерации →
  отдать с пометкой. Утверждение нельзя сделать канон-чистым → `<!-- TODO ред: … -->` (как сейчас в writer).
- `tune-learn`: нет сайдкара/pristine → ошибка + подсказка `--pristine`. Прозы-правок нет → «нечего выводить».
  Regex ломает тест либо даёт FP на хорошем тексте → этот патч **не применять**, показать провал; агент
  сужает паттерн или роняет правку в скилл-предложение.

## Тестирование

- `tests/test_tune_diff.py` — фикстура-пара before/after → ожидаемый триаж (FN/FP/neutral + `rule_hit`).
- Самопроверка словаря — рантайм-доказательство предлагаемых правил (не коммит-тест).
- Шаг применения `tune-learn` переганяет весь `pytest` (регресс-гейт).

## Вне области (YAGNI)

Корпус/состояние между прогонами; ансамбль субагентов; авто-коммит/рендер; пороги авто-промоушена по
повторяемости; итеративное со-письмо (генерация — один проход + само-линт). Не-русская проза. Изменение
механики docx-импорта.

## Открытые вопросы

- Выравнивание прозо-блоков в `tune_diff.py` при крупных перестановках абзацев — `difflib` по блокам может
  дать шумные `replace`; если на реальных правках шумит, перейти на якоря по неизменным предложениям. Решать
  по первым прогонам, не усложнять заранее.
- Стабильность tmp-пути между сессиями (`tune-draft` и `tune-learn` могут идти в разных сессиях) — решено
  derived sibling-именами + `--pristine` override.
